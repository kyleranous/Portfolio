Home Automation

Part 5: Creating Out Own Microservice

INPROCESS

I am the world's OKest cook. Outside of some trusty blue box mac and cheese, or throwing meat over fire, I have to look at a recipe to have any chance at making something edible. But recipe websites are...annoying. You have to scroll through paragraphs of why butter nut squash reminds the author of the color purple in the fall. It's annoying, and it slows me down. Home Assistant has a lot of great integrations, but I havn't found one for storing recipies.

Home Assistant is, at its core, a hub for microservices. Each integration can be thought of as a microservice. I can't find a microservice to fit my culinary needs, so I will make one. We are going to be adding two microservices to our setup here, we are going to add a container with a MySQL database, and we are going create an API using Python and Flask to send information to and recieve information from the database. We will then create a custom UI (Integration) in Home Assistant to act as the interface for our API.

So why are we doing it this way instead of just creating the UI in flask and embedding a webpage like we did with the visualizer in part 3? I want to be able to share information from the recipe book with other integrations. For example, I want to be able to add ingredients from the recipie straight to my shopping list integration. I would also like to add a recipe to my calander to help with meal planning. We could find a recipe, then schedule it on our calander. If we store the expected prep and cook times with the recipe, then we can schedule what time we want to eat, and have Home Assistant back schedule for prepping and cooking.

This tutorial is going to cover the integration of the API and Database into our cluster. I have provided links to my Recipie Book repo and Docker Image. I will not be going into detail in writing APIs in flask. If you would like to learn more about flask, I suggest Miguel Grinberg's Flask Mega Tutorial
